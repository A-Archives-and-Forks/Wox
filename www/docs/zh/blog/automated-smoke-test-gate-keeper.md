# 自动化 Smoke Test：AI 时代最后的 Gate Keeper

Wox 的开发速度一旦提起来，风险模型也会跟着变。

跨平台桌面软件本来就有很多耦合点: Go 后端、Flutter 桌面 UI、插件宿主、设置、存储，以及各个平台自己的行为差异。进入 AI 时代之后，代码生成、重构和试错的速度进一步提升，研发效率确实更高了，但这也意味着另一件事: 看起来“局部合理”的改动，更容易把回归问题带进真实运行链路。

所以我们决定给 Wox 建一套自动化 smoke test 机制。对今天的软件开发来说，自动化测试已经不只是提效工具，它更像是最后一道 gate keeper，用来阻止明显的回归问题直接流到用户手里。

## 为什么要做这件事

单元测试当然有价值，但在发布前，我们更关心的问题其实是另一类:

Wox 能不能真的启动起来，连上后端，显示主窗口，响应输入，打开设置，并且让最关键的几个用户路径稳定跑通？

对 Wox 来说，很多可靠性问题都出现在边界上:

- Flutter UI 是否真的连上了 `wox.core`
- 端口、文件目录和用户目录是否初始化正确
- 桌面窗口是否能按预期展示和隐藏
- 设置页和导航入口在重构后是否还可达

这类问题很难只靠代码审查发现。尤其是在 AI 帮助我们更快地产出代码之后，风险并不会自动消失，只会更快地移动到集成边界上。

## 我们现在这套 Smoke Test 做了什么

目前这套机制落在 `wox.test/` 下面，本质上是一套跨平台的端到端编排流程。

它做的事情并不复杂，但非常实用:

1. 先拉起一个真实的 `wox.core` 开发实例。
2. 强制它使用隔离的 Wox 数据目录和用户数据目录。
3. 通过 `/ping` 等待后端真正 ready。
4. 再跑 Flutter 桌面端的 `integration_test`。
5. 把日志和测试产物完整落盘，方便失败后追查。

这套流程回答的是一个很核心的问题: Wox 在真实启动路径下，到底还能不能工作。

## 为什么这些实现细节很重要

`wox.test/bin/run.dart` 里有几个关键设计，决定了这套 smoke test 能不能真的成为 gate keeper，而不是一个“看起来有测试”的摆设。

首先，每次运行都会创建独立的时间戳产物目录。失败之后，`core.log`、`flutter_test.log`、隔离出来的 `wox-data/` 和 `user-data/` 都会被保留下来，问题可以复盘，而不是只能猜。

其次，runner 会通过 `WOX_TEST_DATA_DIR` 和 `WOX_TEST_USER_DIR` 强制重定向目录，确保 smoke test 不会污染开发者本地正在使用的环境。测试和真实使用环境混在一起，结果一定不可信。

再往下，它会优先使用常见的开发端口 `34987`，如果被占用了，就自动回退到一个空闲端口。同时它不会拍脑袋假设后端“应该已经启动好了”，而是明确轮询 `http://127.0.0.1:<port>/ping`，确认服务 ready 之后才继续执行 UI 测试。

另外，smoke 跑的时候会关闭 telemetry，并且在 Windows 下主动检查是否存在正在运行的 Flutter 开发 UI，避免测试过程被已有进程干扰。

这些细节看起来不大，但它们决定了这套机制是不是具备三个最基本的特性:

- 隔离
- 可重复
- 出问题后可诊断

没有这三点，测试就很难成为可靠的门禁。

## 首批用例覆盖了哪些关键路径

目前的核心 smoke case 放在 `wox.ui.flutter/wox/integration_test/launcher_smoke_test.dart` 里，重点不是“测得很多”，而是先兜住最容易影响用户感知的主链路。

现阶段覆盖的内容包括:

- 主窗口能否正常启动并显示 Launcher UI
- 查询框能否展示并保持可用
- 搜索结果上的键盘上下导航是否正常
- 设置页能否从 Launcher 入口打开
- `general`、`ui`、`data` 等几个基础设置页是否还能正常切换和关闭
- 主题设置、数据备份入口、usage/about 页面这些关键入口是否依然可达

这不是一套“全面测试”，而是一套“高价值、低歧义”的健康检查。Smoke suite 的目标不是覆盖一切，而是尽快发现会挡住用户主路径的回归问题。

## 为什么在 AI 时代它更像最后一道门

AI 让写代码变便宜了，也让“看起来没问题”的代码变便宜了。

这会把工程纪律的重点，从“能不能写出来”逐步推向“我们凭什么相信这次改动真的安全”。

当生成、改写、试验都越来越快之后，最后那道质量关不能更模糊，只能更具体。对我们来说，这道具体的门就是自动化 smoke test:

- 它验证的是行为，而不是口头上的预期
- 它覆盖的是最容易出问题的集成边界
- 它能在合并和发布前给团队一个可重复的信心检查
- 它在失败时会留下足够多的上下文，而不是只给一句模糊的报错

换句话说，AI 可以帮助我们把速度拉起来，但自动化测试决定这份速度是不是安全的。

## 后面还会继续做什么

现在这套 smoke flow 主要覆盖的还是“真实后端 + Flutter 桌面集成测试”这一层，这已经能提供一个很有价值的可靠性基线，但它显然不是终点。

后续很自然的方向，是把同样的思路继续扩展到打包产物验证，以及更多关键可靠性场景上。同时我们也会刻意控制复杂度，避免把它做成一套看起来很庞大、实际上没人愿意跑也没人信的系统。

我们想要的不是“测试很多”。

我们想要的是: 当这套 gate keeper 放在最后一道门上时，团队真的相信它挡住的是该挡住的问题。
